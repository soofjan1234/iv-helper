## Map 并发安全吗？

**结论：不安全。**

- **竞态条件**：多个 Goroutine 同时读写会触发 fatal("concurrent map writes")
- **检测机制**：Map 内部有一个 writing 标志位。任何写操作开始前会检查该位，并将其设为“正在写入”。如果此时发现已经被设为“正在写入”，直接抛出 `fatal error`。
- **解决方案**：
    1. **`sync.Mutex`**：最通用，写锁保护。
    2. **`sync.RWMutex`**：读多写少场景，性能更好。
    3. **`sync.Map`**：官方提供的并发版 Map，适合“读多写少”且“键不怎么变化”的特定场景。
    4. **分段锁**：将一个大 Map 拆分成多个小 Map，减少锁竞争（类似 Java 的ConcurrentHashMap 原理）

## Sync.Map

行为类似 map[any]any，但**多 goroutine 并发读/写/删不需要你再加锁**

**不要默认用 sync.Map**

多数代码应该用普通 map + 自己的锁（或其它同步**）**，这样类型更安全、也方便维护和 map 一起的其它变量

**适合的两种场景**

| 场景 | 说明 | 典型例子 |
| --- | --- | --- |
| **一写多读** | 同一个 key 只写一次，之后大量读 | 只增不减的缓存、配置表 |
| **多 goroutine 操作不同 key** | 各 goroutine 访问的 key 集合几乎不重叠 | 每个 goroutine 维护自己的 key，如 per-goroutine 状态 |

### 结构理解

首先是字典树trie，以空间换时间，能在一组字符串里快速的进行某个字符串的匹配

在源码里，我们用哈希值作为key，一层用 4 位，所以深度 = 哈希位数 / 4。

如果是64位，则会有16个孩子，最多16层深

**再小树太深、Load 会慢很多；再大节点太大、收益几乎可以忽略**

```go
type Map struct {
	_ noCopy
	m isync.HashTrieMap[any, any]
}

type HashTrieMap[K comparable, V any] struct {
// 用于初始化
	inited   atomic.Uint32
	initMu   Mutex
// 哈希 trie 的根节点（内部节点，有 16 个子槽）。用原子指针保证并发读/写根时无需锁住整个 map。
	root     atomic.Pointer[indirect[K, V]]
// 从 key 得到 uintptr 哈希的函数
	keyHash  hashFunc
// 比较两个 value 是否相等的函数，用于 CAS 等操作
	valEqual equalFunc
	seed     uintptr
}

const (
	// 每层用 4 位哈希，子节点数 16；再小负载性能损失大，再大收益很小（约 1%）。
	nChildrenLog2 = 4
	// 每层子节点个数，即 1<<4 = 16
	nChildren     = 1 << nChildrenLog2
	// 取槽位索引用，(hash>>shift)&mask 得到 0..15
	nChildrenMask = nChildren - 1
)

// indirect 是哈希 trie 的内部节点，每层按 4 位选一个子节点往下走。
type indirect[K comparable, V any] struct {
	node[K, V]
	// 删除/收缩时标记本节点已废弃，避免并发下重复操作
	dead atomic.Bool
	// 保护本节点的 children 以及所有子 entry 的修改
	mu Mutex
	// 父节点，用于删除时向上收缩
	parent *indirect[K, V]
	// 16 个子槽位，每个指向 node（可能是 indirect 或 entry）
	children [nChildren]atomic.Pointer[node[K, V]]
}

// entry 是哈希 trie 的叶子节点，存一个 key-value，冲突时用 overflow 链表。
type entry[K comparable, V any] struct {
	node[K, V]
	// 哈希冲突时挂成链表，同一 4 位路径下多个 key 共用一个槽
	overflow atomic.Pointer[entry[K, V]]
	key      K
	value    V
}

// node 是节点的公共头，通过 isEntry 区分实际是 entry 还是 indirect。
type node[K comparable, V any] struct {
	// true 表示实际是 entry（叶子），false 表示是 indirect（内部节点）
	isEntry bool
}

```

### 查找流程

1. 计算hash值
2. 每四位进行匹配槽位
3. 看类型
    1. 是nil，说明key不存在
    2. 是indirect，继续找
    3. 是entry，比对key，匹配则返回，否则沿着overflow找

### 添加流程

1. 跟查找流程一样找插入/更新点
2. 锁住当前层 indirect，再读一次槽和 i.dead
    1. 若槽已被改成 indirect 或节点已死，就解锁并从头重试
3. 持锁后
    1. 槽里已有 entry，更新
    2. 槽是空的，新增
    3. 槽里有 entry 但 key 不在链里，需要expand

### 扩展流程

同一个槽里已经有一个 entry（oldEntry），现在要插入的 key 的 hash 前缀和它一样，但 key 不同，就要在这个槽上建一层（或多层）indirect，把两个 key 按 hash 的更低位分到不同槽

1. 两个 key 的完整 hash 相同（真冲突），另一个挂到 overflow 上
2. 两个 key 的 hash 不同（可以在更低位区分），用 hash 的更低位的 4 位，让 old 和 new 落在不同槽。
