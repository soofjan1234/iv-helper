// ----资料----
MySQL，一个**单进程多线程**架构的数据库

数据库：文件，例如 MYD、MYI、ibd 结尾的文件 

实例：由后台线程以及一个共享内存池组成。用于真正操作数据库文件

### MyISAM 和 InnoDB 的区别

| 特性 | MyISAM | InnoDB |
| --- | --- | --- |
| 事务支持 | 不支持 | 支持 |
| 锁机制 | 表锁 | 行锁 |
| 外键支持 | 不支持 | 支持 |
| 崩溃恢复 | 不支持 | 支持（WAL + Redo Log） |
| MVCC | 不支持 | 支持 |
| 索引结构 | 索引和数据分离 | 索引即数据（聚簇索引） |

# 概念

数据库底层的一套“数据管理插件”，它来决定数据如何存，如何用索引，如何增删改查

| **维度** | **MySQL 实例 (Instance)** | **InnoDB 存储引擎 (Engine)** |
| --- | --- | --- |
| **本质** | 操作系统中的一个**进程** | 数据库的**存取技术实现** |
| **组成** | 连接池 + 解析器 + 优化器 + **存储引擎** | 缓冲池 + 后台线程 + 事务日志 |
| **关系** | **容器**（它是引擎的宿主） | **组件**（它是实例的灵魂） |
| **灵活性** | 相对固定 | **可插拔**，可以随时换一个 |

## 线程

1. Master Thread：将缓冲区的数据异步刷新到磁盘
2. IO Thread：在InnoDB中大量使用AsyncI O来处理写IO请求，IO Thread来负责这些请求的回调
3. Purge Thread：事务被提交后，undolog可能不再需要，Purge Thread来回收
4. Page Cleaner Thread：脏页的刷新操作

## 内存

**缓冲池**

InnoDB是基于磁盘存储的，并将其中的记录按照 **页** 的方式进行管理

但是读磁盘慢，所以有缓冲区 是一块内存区域 提高读性能 

读的时候 放到缓冲区 下次读先读缓冲区；改的时候 优先改缓冲区 以合适的时机（ Checkpoint）写入磁盘

页是 InnoDB 磁盘管理的最小单位，也是内存交换的最小单位。

如果只是一行数据行搬来搬去，磁盘太累，InnoDB就说：不如直接搬16KB进内存，这16KB就叫页

缓冲池中缓存的页有：索引页、数据页、undo页、插入缓冲等等

**LRU、FreeList、FlushList**

缓冲池的空间是有限的，如何高效管理这些“页”呢

1. Free List：记录缓冲池中哪些页是**空的**，还没装数据。当 InnoDB 需要从磁盘读一个新页进入内存时，它先去 `Free List` 找：
    1. **有空位**：直接从链表里取一个空页，装入数据，然后把这个页从 `Free List` 删掉。
    2. **没空位**：说明满了，必须去 `LRU List` 里踢走一个老家伙。
2. LRU List（最近最少使用链表）：管理已经在使用的页，按“活跃度”排序
    1. 有个 **Midpoint（中点）**，把链表分成了“热区”和“冷区”
3. Flush List：专门记录那些被修改过的页（**脏页**）
    1. 当你在内存里改了数据，这个页不仅在 `LRU List` 里，还会被挂到 `Flush List` 上。
    2. **后台线程（Page Cleaner Thread）** 会盯着这个表，定时把这些页刷回磁盘。

关于Midpoint：

假设缓冲区已有活跃的热点数据，我扫描全表，按传统的LRU逻辑，新读入的页全部塞进去，之前的热点数据都没了，又得回磁盘读

Midpoint 把 `LRU List` 拆成了两段（默认比例是 **5:3**）：

- **New Sublist（热区/Young区）**：靠近头部，存放经常访问的页。
- **Old Sublist（冷区/Old区）**：靠近尾部，存放刚从磁盘读入的页。

数据的“晋升”流程：

1. **初次见面（入冷区）**：任何从磁盘读入的新页，**放在 Midpoint 位置**（即冷区的开头）。
2. **观察期**：这个页会在冷区待一会儿。
3. **转正（进入热区）**：如果这个页在冷区待的时间超过了设定的阈值（`innodb_old_blocks_time`，默认 1 秒），并且**再次被访问**，它才有资格晋升到 **New Sublist 的头部**。

**重做日志缓存**

用于redo log

- **修改数据**：数据页在 Buffer Pool 里变脏。
- **生成日志**：同时生成一条 Redo 记录，塞进 **Redo Log Buffer**。
- **定时刷盘**：满足以下三个条件之一，Redo Log Buffer 里的东西就会被甩到磁盘上：
    - 每秒钟刷新一次。
    - 事务提交时（根据配置决定）。
    - Log Buffer 空间剩下一半时。

**额外的内存池**

InnoDB 内部有很多“小零件”需要内存。比如，每个页的控制块（记录页的状态、锁信息、LRU 位置等）、各种数据结构。

# Checkpoint

作用：

1. 缩短数据库的恢复时间
    1. 数据库崩溃重启时，并不是从 redo log 的开头开始重放，而是从最近一次 checkpoint 的位置开始
    2. checkpoint之前，对应的数据页都已经写入磁盘
2. 缓冲池不够用时，将脏页刷新到磁盘
    1. 上文提到的一种情况，FreeList不够用，踢走LRUList的老家伙
    2. 而这些老家伙如果已经修改，就得刷新到磁盘
3. 重做日志不可用时，刷新脏页
    1. redo log是循环使用的，日志写满时，新日志必须覆盖旧日志
    2. 如果旧的没有落盘，会导致数据丢失

LSN （Log Sequence Number）是 redo log 的递增序号，每条修改都会推进它。当数据库声明“checkpoint 在 LSN = X”时，实际含义是：**所有小于等于 X 的修改，其数据页都已经在磁盘上**

# Master Thread

负责很多**维持数据库长期稳定运行的后台工作**

本质上是一个“大循环调度器”：线程每秒醒来做一轮轻量维护，每 10 秒做一轮更重的维护，如果检测到系统空闲，还会进入“后台强化清理模式”。

它负责的事情其实不复杂，主要包括四类：把 log buffer 刷到磁盘、刷脏页以推进 checkpoint、合并 change buffer（旧称 insert buffer）、以及 purge 已经不再需要的 undo 版本。

# 关键特性

## **插入缓冲**

### 1. 为什么需要它？

在 InnoDB 中，**主键**因为通常自增，所以是**顺序**往磁盘写，效率很高。而**二级索引，**因为值通常是离散的，所以是**随机**往磁盘写，影响性能。

### 2. 工作原理

当一个二级索引页不在内存（Buffer Pool）中时，InnoDB 不会立刻去磁盘读取该页，而是先记录下这个修改。

1. **判断条件**：如果要更新的索引是**非唯一**的二级索引，且索引页不在内存里。
2. **存入缓冲**：将修改记录在 `Insert Buffer` 对象中（实际上它是物理存在于共享表空间里的一棵 B+ 树）。
3. **合并（Merge）**：等以后该索引页因为其他查询被读取到内存时，或者后台线程定期扫描时，再将 `Insert Buffer` 里的修改合并到真正的索引页中。

> **注意：** 为什么必须是“非唯一”索引？
因为如果是唯一索引，插入时必须立即去磁盘检查是否重复（Unique Check），这就不得不产生随机 I/O，缓冲也就失去了意义。
> 

### 3. 从 Insert Buffer 到 Change Buffer

从 MySQL 5.5 版本开始，Insert Buffer 得到了升级，现在它叫 **Change Buffer**。因为它不再只针对 `INSERT`，还支持：插入、删除缓冲（标记删除）、物理删除缓冲

## 两次写

### 1. 核心危机：页断裂（Partial Page Write）

InnoDB 的数据页默认是 **16KB**，而操作系统底层（以及磁盘）最小的写入单位通常是 **4KB**。

当 InnoDB 正在往磁盘写一个 16KB 的数据页时，如果写到一半，服务器突然**断电**了：

- 磁盘上的这个页：一半是新的，一半是旧的。也就是**损坏**了（CRC 校验失败）。

> **致命点：** 这种损坏，单纯靠 **Redo Log（重做日志）** 是救不回来的。因为 Redo Log 记录的是“在某个页的偏移量处做了什么修改”，如果这个页本身物理结构都碎了，Redo Log 根本无处下手。
> 

---

### 2. 两次写的工作原理：先留底，再干活

为了防止“页断裂”，InnoDB 增加了一个中间层。Doublewrite由两部分组成，一部分是内存中的Doublewrite Buffer，2MB；另一部分上磁盘上共享表空间连续的128个页，总共2MB

1. **内存缓冲**：当脏页（修改过的页）准备刷新到磁盘时，先不直接写到数据文件里。
2. **第一步（写缓冲区）**：先将脏页复制到内存中的 `Doublewrite Buffer`。
3. **第二步（写共享表空间）**：*这里是顺序写，开销很小。*
4. **第三步（离散写）**：确认第一步写安全后，再把页写入真正的 **.ibd 数据文件**。

---

### 3. 如果发生崩溃，如何自救？

- **情况 A：** 如果在写“共享表空间”时断电，数据页还没动，直接用该页+redolog重做。
- **情况 B：** 如果在写“正式数据文件”时断电，导致页损坏。会去 **Doublewrite Buffer 的共享表空间**里找到那个完整的副本。

## 自适应哈希索引

### 1. 为什么需要它？（B+ 树的局限）

虽然 B+ 树已经很快了，但它毕竟是一棵树。

- 如果你的树有 4 层，查找一行数据通常需要 **4 次逻辑读**（在内存里跳跃 4 个页面）。
- 在极高并发的场景下，即便页面都在内存（Buffer Pool）里，这种反复的“寻道”和“页内比较”也会产生 CPU 开销和锁竞争（Latch 竞争）。

---

### 2. 它是怎么工作的？（“自适应”的精髓）

**AHI 不用你手动创建**（不像你写的 `CREATE INDEX`），它是 InnoDB 观察你的行为后**偷偷干的**。

1. **观察（Monitoring）**：InnoDB 会监控表上的查询。如果它发现某个索引页被频繁访问，而且访问的模式（查询条件）总是固定的。
2. **建立哈希**：它会根据这个索引页的内容，在内存里自动建一个哈希表。哈希表的 **Key** 是索引键值，**Value** 是该索引页在内存中的具体地址。
3. **走捷径**：下次同样的查询再来时，InnoDB 不再从 B+ 树根节点一层层往下爬，而是直接查哈希表，**一跳到位**，直接拿到目标页。

---

### 3. 为什么不给所有数据都建哈希？

哈希索引虽然快，但它有两个致命弱点，这也是为什么它必须是“自适应”的：

- **只支持等值查询**：哈希只能找 `WHERE ID = 100`，没法处理 `WHERE ID > 100`（范围查询）。
- **浪费内存**：哈希索引存在 Buffer Pool 里，会占用宝贵的内存空间。

所以，InnoDB 只会对**热点数据**中**经常等值查询**的部分建立哈希。如果热点变了，它会自动删除旧的哈希索引，建立新的。

## 异步io

InnoDB 发出请求后，**直接转身去干别的事了**。

- **批量提交**：它可以一口气发出 10 个读取请求（比如 10 个不同的索引页）。
- **内核处理**：操作系统层面的 AIO 负责把这些请求排队、优化顺序，然后统一处理。
- **回调机制**：等到磁盘数据准备好了，再通知 InnoDB 来处理。

### 核心优势：

- **提高并发性**：CPU 不会被磁盘的慢速度给“拖垮”。
- **I/O 合并（IO Merge）**：这是 AIO 最牛的地方。如果 InnoDB 发现要读第 10、11、12 号页，AIO 发现它们在磁盘上是连续的，它会把这 3 个请求合并成一个“大请求”，一次性读出 48KB。磁头只用动一次，效率翻倍。

## 刷新邻接页

当 InnoDB 决定把内存里的一个脏页（修改过的页）**刷新到磁盘**时，它会顺便看一眼：**“哎？隔壁的页面是不是也是脏页？”**

- 如果发现邻居页（在同一个区/Extend 内）也是脏的，那就**顺便一起刷掉**。

### 为什么要这么做？

- **机械硬盘（HDD）的福音**：对于机械硬盘来说，磁头移动到某个位置是很费劲的。既然已经停在这个位置了，顺便把旁边的活儿都干了（顺序写），比以后再专门跑回来写一次（随机写）要快得多。

### 现在的尴尬：SSD 时代

到了 SSD（固态硬盘）时代，物理寻道时间几乎为零，随机写的性能也大幅提升。

- **副作用**：如果你开启“刷新邻接页”，可能会把一些原本不需要立刻刷新的页给刷了，反而增加了总写入量，而且 SSD 并不怕随机写。
- **建议**：
    - 如果你用的是 **HDD**，建议开启。
    - 如果你用的是 **SSD**，建议关闭该参数（`innodb_flush_neighbors = 0`）。
