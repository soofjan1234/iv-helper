## 二进制日志（binary log）

属于 **MySQL Server 层。**记录了对MySQL执行更改的所有操作

作用：恢复；主从复制

binlog日志有三种格式，分别为STATMENT、ROW和MIXED。
1. STATEMENT（基于语句的复制）
    优点：日志量极小，节省磁盘空间和网络带宽。
    缺点：容易导致主从数据不一致。比如NOW（）
2. ROW（基于行的复制）
    优点：最安全、最可靠。
    缺点：日志量巨大。
3. MIXED（混合模式）
    默认使用 STATEMENT 模式以节省空间；但当检测到某些 SQL 可能会导致主从不一致时（如使用了 NOW()、UUID() 等），系统会自动切换为 ROW 模式。

## 重做日志文件（Redo Log）

### 1. 物理结构

在 MySQL 的数据目录下，你通常会看到名为 `ib_logfile0`、`ib_logfile1` 这样的文件。它们就是 Redo Log 的实体

InnoDB 的 Redo Log 文件不是一个，而是**一组**（默认 2 个），大小固定。

**两个指针**：

- **write pos（当前写位置）**：随着你不断执行 SQL，这个指针像磁头一样往后挪，记录日志。
- **checkpoint（检查点）**：这是当前已经刷到磁盘数据文件（`.ibd`）的最新的数据页对应的日志位置。在这个位置之前的日志，说明数据已经安全落盘，**可以被覆盖**了。

> **擦除逻辑**：当 `write pos` 追上 `checkpoint` 时，说明“跑道”满了。此时 MySQL 会陷入停顿（抖动），拼命地把内存里的脏页刷往磁盘，好把 `checkpoint` 往后挪，腾出空间写新日志。
> 

### 2. 如何记录

Redo Log 记录的是**物理修改**，它的单位不是“行”，而是 **Log Block（日志块）**。

每个 Block 是 **512 字节**（正好对应磁盘一个扇区的大小，保证写入是原子的）。它内部记录的信息大概长这样：

> “Space ID: 10, Page Number: 45, Offset: 120, Value: 0xAAAA”
> 

这种记录方式比 Binlog（记录 SQL 语句）快得多，因为恢复时，InnoDB 只需要拿到这个页，直接在对应的偏移量覆盖字节即可，**不需要解析 SQL，不需要建立执行计划**

### 3. 为什么写日志快，写数据慢？

既然都要写磁盘，为什么不直接写数据文件，非要先写日志？”

这就是 **WAL 技术（Write-Ahead Logging，预写日志）** 的核心：

- **写数据（慢）**：数据页分布在磁盘各个角落，是**随机 I/O**。
- **写日志（快）**：Redo Log 是追加写的，磁头不动，顺着轨道一直写，这叫**顺序 I/O**。

### 4. 日志的生命周期（从内存到磁盘）

Redo Log 也不是直接下磁盘的，它中间有个缓冲：

1. **Redo Log Buffer**：在内存里，事务产生的日志先存这。
2. **刷盘策略 (`innodb_flush_log_at_trx_commit`)**：
    - **值为 1（最安全）**：每次事务提交都必须实打实写到磁盘。
    - **值为 0**：每秒刷一次。
    - **值为 2**：提交时写到系统缓存，每秒再刷磁盘。

### 5. 崩溃恢复（Crash Recovery）的过程

当数据库意外宕机重启时，它会第一时间去看 Redo Log：

1. **检查 LSN（Log Sequence Number）**：LSN 是一个不断增长的数字，记录了日志的总量。
2. **对比数据页 LSN**：每个数据页（Page）头部也存了一个 LSN。
3. **查漏补缺**：如果日志里的 LSN 是 1000，而磁盘数据页里的 LSN 是 800，说明有 200 个字节的修改在宕机时丢了。
4. **重做（Redo）**：把这 200 个字节的修改直接按日志里的位置填回去。

### 6. 两阶段提交

### 三个步骤

为了保证这两个日志“要么都成功，要么都失败”，MySQL 模拟了分布式事务的 2PC：

### 第一步：Prepare 阶段（准备）

1. InnoDB 引擎将更新记录到 **Redo Log** 中。
2. 将 Redo Log 的状态标记为 `prepare` 状态。
3. 此时还没写 Binlog。

### 第二步：写 Binlog

1. MySQL Server 层将具体的 SQL 或 Row 记录写入 **Binlog**。
2. 将 Binlog 刷入磁盘（依据 `sync_binlog` 配置）。

### 第三步：Commit 阶段（提交）

1. 最后，InnoDB 将 Redo Log 的状态由 `prepare` 改为 `commit`。
2. 事务正式完成。

### 恢复逻辑

如果在执行过程中突然断电，MySQL 重启后会扫描 Redo Log 并检查：

- **Case 1：Redo Log 状态是 `commit`**
    - 直接提交，因为 Binlog 肯定也写成功了。
- **Case 2：Redo Log 状态是 `prepare`**
    - 这时会去检查 **Binlog**。
    - **如果 Binlog 存在对应的记录**：说明是在第 3 步断电的，会自动把 Redo Log 设为 `commit`（恢复数据）。
    - **如果 Binlog 不存在**：说明是在第 2 步写 Binlog 前断电的，会**回滚**这个事务。


## Redo Log vs Binlog

| 对比项 | Redo Log | Binlog |
| --- | --- | --- |
| 作用 | Crash Recovery（崩溃恢复） | Point-in-time Recovery（时间点恢复） + 主从复制 + 审计 |
| 层级 | InnoDB 引擎层 | Server 层（支持所有引擎） |
| 内容 | 物理日志（基于 Page） | 逻辑日志（SQL/数据/混合） |
| 写入时机 | 多元 | 事务提交时 |
