InnoDB存储引擎仍然支持B+树索引、全文索引、哈希索引。哈希索引是自适应的，不能够人为干预是否生成。B+数不能够给到一个具体的值，它只能查找数据行所在的页。

### 为什么用 B+ 树？

| 数据结构 | 问题 |
| --- | --- |
| Hash | 不支持顺序和范围查询 |
| 二叉查找树（左<右） | 可能退化成链表，O(n) |
| 平衡二叉树 | 维持平衡频繁旋转；范围查询，磁盘IO 次数多 |
| 红黑树 | 大致平衡；多次IO |
| B树 | 所有节点存放kv |
| B+ 树 | 叶子节点存放kv，并有链表，有助于范围查询 |

**面试话术（结论先行）**：  
数据库索引要兼顾**等值、范围、排序**，且数据在磁盘上，希望**树矮、IO 少**。  
- **Hash**：只能等值，不支持范围、ORDER BY，所以不适合做主索引。  
- **二叉/平衡/红黑**：树高太大，范围查询要多次跨层 IO；且平衡树旋转代价高。  
- **B 树**：非叶节点也存数据，单页能挂的键少，树更高，IO 次数多。  
- **B+ 树**：非叶节点只存键和指针，单页能放更多键，树更矮；叶子成链表，范围查询顺序读即可，所以 InnoDB 用 B+ 树做索引。

## B+树

B+ 树由两种“页”组成：

- **非叶子节点（目录页）**：只存**索引键值**和**指向子页的指针**。
- **叶子节点（数据页）**：存放真正的**整行数据**

### 插入

1. Leaf Page没满，直接插入Leaf Page
2. Leaf Page满，Index Page没满
    1. 拆分Leaf Page
    2. 中间的节点放Index Page
3. 两个都满
    1. 拆分Leaf Page
    2. 小于中间的放左边，大于的放右边
    3. 拆分上一层节点，也就是Index Page
    4. 小于中间的放左边，大于的放右边
    5. 中间节点放入上一层IndexPage（倒数第二层

分裂要申请新页，改动父节点的索引项，故b+树同样提供了旋转操作

当Leaf Page满了，但左右兄弟节点没满，优先查看左兄弟

### 删除

B+树用填充因子来控制删除，50%是填充因子可设的最小值

1. 删除一条记录后，填充依然在50%以上
    1. 直接删除
    2. 如果该页还是IndexPage的节点，用该节点的后继节点代替
2. 删除后，该IndexPage的填充<50%，兄弟节点>50%
    1. 旋转，从兄弟那“借”一个键值过来，调整一下父节点的索引值
3. 删除后，都<50%
    1. 合并叶子节点和兄弟节点
    2. 更新IndexPage

## B+树索引

| 类型 | 说明 |
| --- | --- |
| 聚簇索引 | 叶子节点存放的是完整的每行的记录 |
| 非聚簇索引 | 叶子节点存主键值，需回表查询 |

## Cardinality值

当你执行 `SHOW INDEX FROM table_name;` 时，你会看到一列叫 `Cardinality` 的值。这个值在 InnoDB 中是**预估值**，不是实时的精确统计。

- **高 Cardinality**：列中的值几乎不重复（例如：主键 ID、身份证号、手机号）。
- **低 Cardinality**：列中的值大量重复（例如：性别、状态位 [0/1]、逻辑删除标记）。

### **它如何生成的？（采样统计）**

InnoDB 不可能每次都去全表扫描来计算基数（那太慢了），它是通过**采样**完成的：

1. 随机取 B+ 树的 n 个叶子节点页（默认是8）。
2. 统计这些页中不同记录的数量，得到平均值。
3. 乘以总页数，得出全表的 Cardinality 预估值。

### **什么时候会更新？**

由于是采样预估，这个值可能会“过期”。InnoDB 触发统计更新的常见时机包括：

1. 表中数据修改（Insert/Update）超过了 1/16的行数。
2. 执行了 `ANALYZE TABLE` 命令（当你发现明明有索引但 SQL 跑得很慢时，通常用这个命令强制刷新基数）。

## B+树索引类型

| 类型 | 说明 |
| --- | --- |
| 主键索引 | 唯一且非空，聚簇索引 |
| 唯一索引 | 保证列值唯一 |
| 普通索引 | 加速查询 |
| 联合索引 | 多列组合，遵循最左前缀 |
| 覆盖索引 | 索引包含所需字段，无需回表 |

### 索引失效场景

1. 对索引列做计算或函数操作
2. 隐式类型转换
3. LIKE '%xx'（左模糊）
4. OR 条件中有非索引列
5. IS NULL / IS NOT NULL
6. 联合索引违反最左前缀原则
7. 优化器判断全表扫描更快


## 哈希与查找

InnoDB 需要管理海量的内存页（Pages），维护了一个全局哈希表，并使用链表法处理冲突

- **Key**：`space_id` (表空间 ID) + `page_no` (页号)。
- **Value**：指向该页在 **Buffer Pool** 中位置的指针。

查找过程

1. 除法散列法：h (k) = k mod m，定位到槽
2. 走链表定位到真正记录/页
3. 如何在 16KB 的页内快速找到某一行？
    - InnoDB 会把页内的记录分成几组（每组 4~8 条）。
    - 每个槽位记录这一组中**最大**那条记录的相对地址。
    - 然后使用二分查找

## 为什么索引比不加、全表快
1. 算法层面
    全表扫描：它是线性查找。如果你有 1000 万行数据，要找 
    1000 万行数据，B+ 树通常只需要 3 到 4 次 磁盘 IO 就能定位到数据
2. 减少磁盘IO的次数
    如果没有索引，它就必须得加载所有的磁盘页到内存里来。
    如果有了索引，它就只用加载索引页到磁盘里来。
