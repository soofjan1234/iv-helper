## 什么是事务

原子性、持久性和隔离性，都是为了保证数据库状态的一致性

| 特性 | 说明 | 实现方式 |
| --- | --- | --- |
| 原子性 (Atomicity) | 事务要么全部成功，要么全部失败 | Undo Log |
| 一致性 (Consistency) | 事务前后数据保持一致 | 应用层 + 数据库约束 |
| 隔离性 (Isolation) | 事务之间互不干扰 | 锁 + MVCC |
| 持久性 (Durability) | 提交后数据永久保存 | Redo Log |

分类：

1. 扁平事务
    1. 由 `BEGIN/START TRANSACTION` 开始，由 `COMMIT` 或 `ROLLBACK` 结束
2. 带有保存点的扁平事务
    1. 在事务执行过程中设置标记 `SAVEPOINT name`
    2. 回滚到保存点后，事务并没有结束，你仍需决定最终是全提交还是全回滚。
3. 链事务
    1. 保存点是易失的
    2. 在提交一个事务时，释放所有锁，并将必要的处理上下文**隐式地传给**下一个事务
    3. 提交操作和开启新事务操作合二为一，有利于节省开销
    4. 因为持久性，所以只能恢复到最近的一个保存点
4. 嵌套事务
    1. MySQL不支持
5. 分布式事务
    1. 当一个事务需要跨越多个数据库（或者多个资源管理器）时，就会用到它。
    2. 比如ATM转账

## 事务的实现

### redo

实现事务的持久性。由两部分组成：内存中的重做日志缓冲（redo log buffer），易丢失；磁盘中的重做日志文件（redo log file），持久的。

**Redo Log 刷盘策略（innodb_flush_log_at_trx_commit）：**

- **0 (延迟刷)**：事务提交不动作。Master Thread 每秒将 `Log Buffer` -> `OS Cache` -> `Disk`。
- **1 (实时刷)**：事务提交即执行 `fsync`。确保日志必须进入磁盘文件（Log Group）。
- **2 (实时写，延迟刷)**：事务提交只写到 `OS Cache`，每秒再 `fsync` 到磁盘。

**其他被动刷盘触发点：**

1. `Log Buffer` 空间使用达 **50%**（强制 Write）。
2. **Checkpoint** 检查点推进（强制 fsync 关联日志）。
3. **Master Thread** 每秒定时刷新。

log block：重做日志是以512字节存储的。为了匹配磁盘扇区（Sector）的大小，可以原子写入

**LSN**

1. 重做日志写入的总量。
    - 记录了当前 `log buffer` 写到了哪里，以及磁盘上的 `ib_logfile` 写到了哪里。
2. Checkpoint 的位置
    - **作用**：在 Checkpoint LSN 之前的 Redo Log 已经完成了使命，可以被覆盖重用了。
    - **灾备意义**：宕机重启时，InnoDB 会直接从 Checkpoint 记录的 LSN 开始往后扫描，前面的就不看了，极大缩短了恢复时间
3. 页的版本
    - 每个数据页头部都有一个 `FIL_PAGE_LSN`，记录了该页最后一次被修改时的 LSN
    - 如果 `页 LSN` < `日志 LSN`：说明该页落后于日志，需要 **Redo**

### undo

用于**事务回滚** 和 **MVCC**

- **逻辑日志**
    - 与 Redo Log 记录某个页某个位置改了什么不同，Undo Log 记录原来值是什么。
- **反向操作**
    - 它执行的是与你做的操作**相反**的指令。比如insert，执行delete。
    - 不是物理直接恢复，因为可能有别的事务在同时修改该页。

**Undo Log 的分类**

1. **Insert Undo Log**：
    - 只在 `INSERT` 时产生。
    - **特点**：因为新插入的行对其他事务不可见（隔离性），所以这个日志在事务提交后就可以**立即删除**。
2. **Update Undo Log**：
    - 在 `UPDATE` 和 `DELETE` 时产生。
    - **特点**：因为它可能要用于 MVCC 读，所以提交后不能立刻删。必须等系统中所有可能用到它的快照读事务都结束了，由 **Purge 线程** 统一清理

### purge

用于处理undo log

1. **真正的物理删除（清理垃圾）**
    1. 将那些打了 `delete-mark` 标记的**记录**从 B+ 树索引中彻底物理删除
    2. 当某条记录被标记了 `delete-mark`，且**全数据库已经没有任何事务**需要看这行的旧版本时，它就会被清理
2. **回收 Undo Log 空间**
    1. Undo Log 不能无限增长。
    2. 当 Purge 确认某条 Undo Log 已经没用了（没有 ReadView 指向它），它会释放 Undo Page 里的空间。让这些空间可以被其他事务重复使用
3. History List。为了让 Purge 找得快
    1. 当事务 `COMMIT` 时，它的 Undo Log 就会被挂在 History List 的末尾。
    2. Purge 线程从 List 的**头部**（也就是最老的日志）开始往后扫。
    3. 只要碰到“还在被需要”的日志，它就停下来。

### Group commit

把这 10 个事务的日志攒在一起，**只调用 1 次 fsync，优化性能。**
