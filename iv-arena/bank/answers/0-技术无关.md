## 3. 公司

### 3.1 公司主要业务是什么

### 3.2 一个需求下来 要展示要后端 你们怎么沟通的

我们这边产品是技术出身，懂得比较多，所以需求阶段就能和技术对齐可行性，不会出现“产品拍板了技术实现不了”的情况。整体是**需求先对齐、再契约、再并行开发**：

**需求与方案阶段**
1. **方向提出**：产品提需求方向或目标（如“要做共享/外网访问”），会带一些技术上的想法。
2. **技术预研/对齐**：技术评估可行性（系统是否支持、有没有坑），因为产品懂技术，沟通成本低，预研结论能快速达成一致。
3. **线稿/UI 评审**：产品出线稿或高保真，技术+产品一起过交互和边界情况，细节定稿后再进入开发。

**开发与联调阶段**
4. **定义契约**：需求确认后，先出 API 文档（或接口约定），作为前后端共识。
5. **接口评审**：拉上前端/客户端确认字段是否够用，避免返工；产品有时也会参与，能听懂接口设计。
6. **并行开发**：提供 Mock 数据，前端不用等后端写完就能开工。
7. **高效联调**：最后联调；因为前面对齐得比较细，返工少。

### 3.3 讲讲你的 AI 工作流？怎么做 Code Review 的？

1. 在写完一个功能后，我会通过skills触发codeReview，然后对问题进行查看，并且修改记录
2. 在其它地方我也有充分利用AI，比如说前期调研，实现方法，文档编写。

### 3.4 对需求有质疑怎么办？设计不合理怎么办？

- **给依据**：不只是说“不行”，而是列出技术风险（性能瓶颈）或用户体验数据（误触率）。
- **给选项**：提供 Option A（原方案代价）和 Option B（低成本替代方案），让产品做选择题
